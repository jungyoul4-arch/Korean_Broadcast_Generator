# 보리스 워크플로우 (Boris Workflow)

Anthropic의 Claude Code 총책임자 보리스(Boris)가 실제로 사용하는 방식을 기반으로 한 Claude Code 최적 활용 워크플로우.
이 스킬은 Claude Code를 10%가 아닌 100% 활용하기 위한 5가지 핵심 원칙과 실전 패턴을 정의한다.

---

## 원칙 1: 계획 먼저, 코딩은 나중에 (Plan Before Code)

코드를 작성하기 전에 반드시 **Plan 모드**로 전체 상황을 분석하고 설계한다.

### 왜 중요한가
"이거 만들어줘"라고 바로 코딩에 들어가면 전체 구조를 놓치고, 중간에 방향을 바꾸느라 시간을 낭비하게 된다. 계획을 먼저 세우면 한 번에 올바른 방향으로 작업할 수 있다.

### 실행 방법

1. **Plan 모드 진입**: 작업 시작 시 `Shift+Tab`으로 Plan 모드 전환
2. **현황 분석**: 기존 코드베이스, 파일 구조, 관련 파일들을 먼저 파악
3. **설계도 작성**: 어떤 파일을 수정/생성할지, 어떤 순서로 작업할지 계획 수립
4. **사용자 확인**: 계획을 사용자에게 보여주고 승인 받은 후 실행

### 고급 패턴: 크로스 검증
- 하나의 Claude Code 창에서 계획을 세운 뒤, **다른 창**의 Claude에게 그 계획을 검토하게 한다
- "이 계획을 선임 개발자 입장에서 검토해줘. 빠진 것, 위험한 것, 더 나은 방법이 있는지 알려줘"
- 이 과정을 통해 계획의 결함을 실행 전에 발견할 수 있다

### 예시 프롬프트
```
# 계획 수립 요청
"로그인 기능을 만들려고 해. 먼저 Plan 모드에서 전체 구조를 설계해줘.
어떤 파일이 필요하고, 어떤 순서로 작업할지 정리해줘."

# 크로스 검토 요청 (다른 창에서)
"다음 계획을 선임 개발자처럼 검토해줘:
[계획 내용 붙여넣기]
문제점이나 개선사항을 알려줘."
```

## 원칙 2: 자기 확인 (Self-Verification)

작업을 마친 후 Claude가 **스스로 결과를 검증**하게 한다.

### 왜 중요한가
코드를 작성하는 것과 코드가 올바르게 동작하는 것은 다르다. 자기 확인 단계를 추가하는 것만으로 결과물 품질이 2~3배 향상된다.

### 실행 방법

1. **실행 확인**: "작성한 코드를 직접 실행해서 에러가 없는지 확인해"
2. **브라우저 확인**: "브라우저를 열어서 화면이 제대로 나오는지 확인해" (Playwright MCP 활용)
3. **테스트 실행**: "관련 테스트를 돌려서 통과하는지 확인해"
4. **타입 체크**: "TypeScript 타입 에러가 없는지 `tsc --noEmit`으로 확인해"
5. **린트 체크**: "린트 에러가 없는지 확인해"

### 검증 체크리스트
작업 완료 시 최소한 다음을 확인한다:
- [ ] 코드가 에러 없이 실행되는가
- [ ] 의도한 기능이 정상 동작하는가
- [ ] 기존 기능이 깨지지 않았는가
- [ ] 콘솔에 경고/에러가 없는가

### 예시 프롬프트
```
"이 기능 구현이 끝나면 직접 실행해서 결과를 보여줘.
에러가 있으면 수정하고, 정상 동작할 때까지 반복해."

"Playwright로 브라우저 열어서 홈페이지가 제대로 렌더링되는지 스크린샷 찍어줘."
```

## 원칙 3: 실수 노트 누적 관리 (Cumulative Mistake Notes)

프로젝트의 **CLAUDE.md** 파일에 규칙, 방향, 자주 하는 실수를 기록하고 지속적으로 업데이트한다.

### 왜 중요한가
Claude Code는 세션 간 기억을 공유하지 않는다. 하지만 CLAUDE.md에 학습 내용을 누적하면, 시간이 지날수록 프로젝트에 완벽히 적응한 '맞춤형 개발자'가 된다. 복리 효과로 점점 더 정확해진다.

### CLAUDE.md 구성 템플릿

```markdown
# 프로젝트 헌법 (CLAUDE.md)

## 프로젝트 개요
- 프로젝트명: [이름]
- 기술 스택: [예: Hono, TypeScript, Cloudflare Workers, D1 SQLite]
- 배포 환경: [예: Cloudflare Pages]

## 핵심 규칙
- [예: Tailwind CDN 사용 시 JIT arbitrary class 사용 금지 → 커스텀 스타일은 별도 CSS 파일에]
- [예: 앱 명칭 구분 — "플랜앱" ≠ "K1", 별개 프로젝트]

## 폴더 구조
- [주요 디렉토리 설명]

## ⚠️ 실수 노트 (Mistake Log)
### 날짜: YYYY-MM-DD
- **실수**: [무엇을 잘못했는지]
- **원인**: [왜 발생했는지]
- **해결**: [어떻게 고쳤는지]
- **교훈**: [다음부터 어떻게 방지할지]

## 작업 컨벤션
- [코딩 스타일, 네이밍 규칙 등]
```

### 실수 발생 시 프로세스
1. 실수를 발견하면 단순히 수정하는 데 그치지 않는다
2. "CLAUDE.md의 실수 노트에 이 실수를 추가해. 원인과 방지책도 함께 적어"라고 지시
3. 다음 세션부터 Claude가 CLAUDE.md를 읽고 같은 실수를 반복하지 않는다

### 예시 프롬프트
```
"방금 만든 실수를 CLAUDE.md 실수 노트에 기록해.
다시는 같은 실수 하지 않도록 교훈도 적어줘."

"CLAUDE.md를 읽고 이전에 기록된 실수들을 확인한 뒤 작업을 시작해."
```

## 원칙 4: 가장 똑똑한 모델 사용 (Use the Smartest Model)

속도보다 정확도를 우선한다. **가장 지능이 높은 모델**을 사용한다.

### 왜 중요한가
빠르지만 덜 똑똑한 모델로 여러 번 수정하는 것보다, 느리지만 정확한 모델로 한 번에 끝내는 것이 전체 시간과 비용을 아낀다. 특히 복잡한 아키텍처 설계나 디버깅에서 그 차이가 극대화된다.

### 실행 방법
- Claude Code 설정에서 **Opus** 모델 선택 (가능한 경우)
- 단순 반복 작업(포매팅, 이름 변경 등)은 빠른 모델도 괜찮지만, **설계·디버깅·복잡한 로직**은 반드시 최상위 모델 사용
- 모델 선택에 아끼지 말 것 — 재작업 비용이 더 크다

## 원칙 5: 세션 전환 시 요약 저장 (Save Session Summaries)

Claude Code는 세션이 바뀌면 이전 내용을 **완전히 잊어버린다**. 반드시 작업 내용을 저장한다.

### 왜 중요한가
컨텍스트 윈도우가 한계에 도달하거나 세션을 종료하면 모든 작업 맥락이 사라진다. 요약을 저장해두면 다음 세션에서 즉시 이어갈 수 있다.

### 실행 방법

1. **작업 중간 저장**: 컨텍스트가 길어지면 "지금까지 한 내용 요약해 줘"
2. **세션 종료 전 저장**: "오늘 작업 전체를 요약해서 `SESSION_LOG.md`에 저장해줘"
3. **다음 세션 시작**: "SESSION_LOG.md 읽고 이어서 작업해"

### 세션 로그 템플릿

```markdown
# 세션 로그 — YYYY-MM-DD

## 완료된 작업
- [작업 1 설명]
- [작업 2 설명]

## 진행 중인 작업
- [현재 상태와 다음 단계]

## 발견된 이슈
- [해결되지 않은 문제들]

## 다음 세션에서 할 일
- [ ] [구체적인 할 일 1]
- [ ] [구체적인 할 일 2]

## 중요 결정사항
- [아키텍처 결정, 기술 선택 등]
```

### 예시 프롬프트
```
"컨텍스트가 길어지고 있어. 지금까지 작업 내용을 SESSION_LOG.md로 저장해줘."

"SESSION_LOG.md 읽고 어제 작업 이어서 해줘."
```

## 보너스 원칙: 영역별 폴더 분리 (Domain Separation)

대규모 프로젝트에서는 **영역별로 폴더를 분리**하고 각 영역에 담당 Claude를 배치한다.

### 실행 방법
- 기능 영역별로 독립적인 폴더 구조 유지 (예: `modules/records/`, `modules/timetable/`)
- 각 영역에 대해 별도의 Claude Code 창을 열어 전문화된 작업 수행
- 영역 간 의존성은 CLAUDE.md에 명시

---

## 멀티태스킹 전략

Claude Code 창을 **여러 개 열어** 동시에 작업을 진행한다.

### 패턴
- **창 1**: 프론트엔드 UI 작업
- **창 2**: 백엔드 API 작업  
- **창 3**: 테스트 작성
- **창 4**: 문서 업데이트

### 알림 설정 (Hooks)
작업 완료 시 소리 알림을 설정하면 각 창의 진행 상황을 놓치지 않고 관리할 수 있다.
Claude Code 설정에서 Hook을 구성하여 작업 완료/실패 시 알림을 받는다.

## 전체 워크플로우 요약

새로운 작업이 들어오면 다음 순서를 따른다:

```
1. CLAUDE.md 읽기 → 프로젝트 규칙/실수 노트 확인
2. Plan 모드 → 작업 계획 수립 (필요 시 크로스 검증)
3. 사용자 확인 → 계획 승인
4. 구현 → 가장 똑똑한 모델로 코딩
5. 자기 확인 → 실행/테스트/린트 검증
6. 실수 발생 시 → CLAUDE.md 실수 노트 업데이트
7. 세션 종료 전 → SESSION_LOG.md 저장
```

이 순서를 매 작업마다 반복하면, Claude Code가 단순한 코딩 도구가 아닌 **체계적인 개발 파트너**로 기능한다.
